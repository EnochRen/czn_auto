# CZN Zero Farm 节点（状态）判断机制说明

本文档总结《卡厄思梦境》自动化框架的**节点（GameState）判断机制**：框架定义了哪些节点、底层如何用图像识别判断"当前在哪个节点"，以及判断到节点后做什么。

---

## 一、整体原理

框架是**纯 OpenCV 模板匹配 + 状态机**，没有任何 AI / OCR 模型参与节点判断（OCR 仅用于赛季刷 Buff 模式的关键词查找，不参与状态识别）。

判断流程一句话概括：

> 截当前游戏画面 → 拿一组预先采集好的"模板小图"在画面里逐个找 → **第一个匹配成功的模板**所对应的节点，就是"当前节点" → 按该节点执行对应点击。

核心代码分布：

| 环节 | 位置 |
|---|---|
| 节点枚举定义 | `detector.py` → `class GameState` |
| 模板匹配算法 | `detector.py` → `class TemplateMatcher` |
| 节点判断逻辑 | `detector.py` → `StateDetector.detect()` |
| 判断结果的动作分发 | `core/gui/worker.py` → `AutomationWorker._dispatch_state()`（主入口）、`main.py` → `handle_state()`（CLI，覆盖较少） |

---

## 二、底层如何"判断当前是什么节点"

### 1. 模板匹配（TemplateMatcher）

- 启动时把 `templates_cn/` 或 `templates_global/`（由 `config.json` 的 `template_profile` 决定）里所有 PNG 加载进内存，**以文件名（不含扩展名）作为模板名**。
- 匹配时把画面和模板都转成**灰度图**，调用 OpenCV：

```python
res = cv2.matchTemplate(frame_gray, tpl_gray, cv2.TM_CCOEFF_NORMED)
_, max_val, _, max_loc = cv2.minMaxLoc(res)
if max_val >= threshold:   # 默认 0.8
    # 命中：返回 (True, 相似度, 模板中心坐标)
```

- **返回的坐标是模板在画面中匹配位置的中心点**，后续点击就基于这个坐标（可叠加 `config.json` 的 `click_offsets` 偏移）。
- `match_all()` 是变体：返回所有命中位置并按 x 从左到右排序，用于"点最左侧那个"等场景（如 `skip_leftmost`）。

### 2. 判断顺序 = 优先级（StateDetector.detect）

`detect()` 内部维护一个**有序的 `checks` 列表**，元素为 `(模板名, 对应GameState)`。它**按顺序逐个尝试匹配，第一个命中的立即返回**，不再往下看。所以：

> **列表中越靠前的模板，优先级越高。**

这一点很关键——它解决了"一个画面可能同时存在多个可识别元素"的冲突。例如撤退按钮 `retreat` 排在最前，`buff_select`、`retreat_btn` 等排在很后，保证关键/独占性强的元素先被认定。

命中后会记录：
- `last_state`：当前节点
- `last_pos`：命中位置中心坐标（动作分发时用来点击）
- `last_template`：具体命中的是哪个模板（同一节点可能由多个模板触发，动作里据此区分）
- `last_conf`：相似度
- `history`：最近 20 次状态历史

### 3. 三个特殊机制

**(a) 差异化阈值**
个别模板用更严格阈值，避免误判（默认 0.8）：

```python
thresholds = {
    "auto_battle_off": 0.95,
    "wrong_page": 0.98,
    "codex_btn3": 0.95,
}
```

**(b) 跳过模板（skip_templates）**
`detect(frame, skip_templates)` 可传入要跳过的模板集合。例如配置 `retreat_on_first_floor=False` 时，会把 `retreat` 加入跳过集，避免一层就误触撤退。

**(c) 一个节点 ↔ 多个模板（多对一）**
许多 `GameState` 由多个不同模板共同映射，提高识别覆盖率（不同画面变体 / 国服国际服差异 / 保底图）。例如：
- `ROOM_SELECT` ← `room_battle` / `room_elite` / `room_event` / `room_rest` / `room_fallback`
- `RESULT_NEXT` ← `result_settle` / `settlement_click` / `node_settlement` / `next_step` / `dismantle_confirm` / `dismantle_equip` / `settlement_confirm`
- `EVENT_SCREEN` ← `event_screen` / `event_option` / `event_option2` / `event_fallback` / `event_fallback2`
- `SELECT_CHARACTER` ← `select_character` / `strong_select` / `unex_select`
- `CODEX_SYNTH` ← `codex_synth` / `codex_btn0~4`

**(d) 兜底（UNKNOWN）**
所有模板都没命中则返回 `GameState.UNKNOWN`。主循环对 UNKNOWN 的处理：
- GUI：累计计数，连续 50 次仅告警（不乱点），继续等待下一帧。
- CLI（`main.py`）：连续 20 次后点击"默认节点"尝试解卡。

### 4. 像素点判断（pixel_checks）

除模板匹配外，框架支持**像素点判断**作为补充手段，用于模板分不清、或没有稳定模板的细分场景。它和模板匹配**共用同一套有序优先级框架**，规则写在独立文件 **`templates_colors/`** 目录里（不写进 `config.json`），启动时由 `detector.py` 的 `load_pixel_checks()` 读取、`StateDetector` 解析。

**规则文件位置（按 `template_profile` 区分国服/国际服）：**

- 优先读 `templates_colors/<template_profile>.json`（如 `templates_colors/templates_cn.json`）；
- 缺失时回退 `templates_colors/pixel_checks.json`；
- 文件内容可为**规则数组**，或含 `"pixel_checks"` 键的对象；不存在/读取失败均按"无规则"处理。

**规则字段：**

| 字段 | 说明 |
|---|---|
| `state` | 命中后认定的 `GameState`（用枚举的字符串值，如 `"combat"`） |
| `points` | 像素点数组，**一个 check 可含多个点**。每点：`rx`/`ry`（相对坐标 0~1）、`rgb`（**RGB 顺序**目标色）、`tol`（可选，每通道容差，默认 10） |
| `mode` | `all`（默认，所有点都命中才算）/ `any`（任一点命中即算） |
| `click` | 可选 `[rx, ry]` 相对坐标，命中后点这里；不写则默认点 `points` 的第一个点 |
| `before` | 可选模板名，把本规则插到该模板判断**之前**；不写则放到所有模板**最前**（优先级最高） |
| `name` | 可选，日志中 `last_template` 显示名；默认 `pixel:<state>` |

**判定逻辑：**

- 相对坐标按 `frame` 实际尺寸换算：`px = rx * frame_w`、`py = ry * frame_h`，因此**天然适配任意分辨率/缩放**，无需关心桌面型/窗口型后端差异。
- 取色时注意 OpenCV 是 **BGR**，配置里写 **RGB**，框架内部已转换。
- 每通道独立容差：`|r-R|≤tol 且 |g-G|≤tol 且 |b-B|≤tol`，默认 `tol=10`（即上下浮动 10）。
- 命中后照常写入 `last_state / last_pos / last_template / last_conf(=1.0)` 并 `return`，动作分发逻辑与模板命中完全一致。

**配置示例（`templates_colors/templates_cn.json`）：**

```json
[
  {
    "state": "combat",
    "points": [
      { "rx": 0.885, "ry": 0.907, "rgb": [230, 60, 60] },
      { "rx": 0.120, "ry": 0.050, "rgb": [255, 255, 255] }
    ]
  },
  {
    "state": "death_screen",
    "mode": "any",
    "before": "main_menu",
    "points": [ { "rx": 0.5, "ry": 0.5, "rgb": [200, 0, 0], "tol": 15 } ],
    "click": [0.5, 0.8]
  }
]
```

相关代码：`detector.py` → `load_pixel_checks()`（从 `templates_colors/` 读取）、`StateDetector._parse_pixel_checks()`（解析校验）、`_match_pixel_rule()`（多点校验）、`detect()`（按 `before` 混入优先级序列）。

## 三、节点（GameState）清单

框架共定义 **60+ 个节点**。下表按**功能分组**列出节点、触发模板及命中后的动作（动作以 GUI `_worker` 为准）。

### 1. 入口 / 准备阶段

| 节点 | 触发模板 | 含义与动作 |
|---|---|---|
| `MAIN_MENU` | `main_menu` | 主界面 → 进入零式系统 |
| `ZERO_SYSTEM_ENTRY` | `zero_system_entry` | 零式系统 → 选法典 |
| `CODEX_SELECT` | `codex_select` | 选好法典 → 确认进入（runs+1） |
| `TEAM_ENTER` | `team_enter` / `team_enter2` / `team_confirm` | 配队 → 进入副本 |
| `SELECT_CHARACTER` | `select_character` / `strong_select` / `unex_select` | 选择角色 → 多点位依次点击 |
| `BUFF_SELECT` | `buff_select` | 进图首次选 Buff（带去重 / 冷却） |
| `DELETE_SAVE` | `delete_save` | 删除存档确认 |

### 2. 地图 / 节点选择

| 节点 | 触发模板 | 含义与动作 |
|---|---|---|
| `MAP_SCREEN` | `map_screen` | 地图界面 → 点默认节点 |
| `ROOM_SELECT` | `room_battle/elite/event/rest/fallback` | 节点选择 → 按 `room_priority` 优先级选房间 |
| `BOSS_NODE` | `boss_node` | Boss 节点 → 左偏移点击 |
| `UNEXPECTED_ROOM` | `unexpected_room` | 误入意外房间 → 离开（`unex_leave`） |
| `CHAOS_CENTER` | `chaos_center` | 前往混沌中心 |
| `CONTINUE_FORWARD` | `continue_forward` / `continue_forward2` | 继续前进 |

### 3. 战斗

| 节点 | 触发模板 | 含义与动作 |
|---|---|---|

| `CODEX_COMPLETE` | `codex_complete` / `codex_complete2` | 完成法典 |
| `CODEX_OBTAIN` | `codex_obtain` | 获得法典 → 两段点击 |
| `CODEX_CONFIRM` | `codex_confirm` | 确认图鉴 |

### 8. 通用确认 / 容错

| 节点 | 触发模板 | 含义与动作 |
|---|---|---|
| `CONFIRM` | `confirm` | 通用确认 |
| `CONFIRM_OPTION` | `confirm_option` | 确认弹窗选项 |
| `DREAM_CONFIRM` | `dream_confirm` | 梦境确认 |
| `CLOSE_VIEW` | `close_view` | 关闭视图 |
| `CLOSE_MISTOUCH` | `close_mistouch` | 关闭误触界面（上移300）|
| `WRONG_PAGE` | `wrong_page`（阈值0.98）| 误入其他页面 → 返回 |
| `BUG_CLOSE` | `bug_close` | 国服 bug 弹窗关闭 |
| `UNKNOWN` | —（无任何模板命中）| 未知状态，等待 / 解卡 |

> 注：`main.py`（CLI 入口）的 `handle_state` 只实现了入口、地图、战斗、事件、休息、死亡、提取等**核心子集**；**完整的 60+ 节点动作分发在 `core/gui/worker.py` 的 `AutomationWorker._dispatch_state` 里**，GUI 是实际主入口。

## 四、判断流程图

```
┌─────────────────────────────────────────────┐
│ main_loop / AutomationWorker.run 每帧循环     │
└───────────────┬─────────────────────────────┘
                │ capturer.capture() 截游戏画面
                ▼
        StateDetector.detect(frame, skip_templates)
                │
                │ 按 checks 列表顺序（= 优先级）逐个：
                │   TemplateMatcher.match(模板, 阈值)
                │   命中 → 记录 last_pos/last_template
                │          立即 return 对应 GameState
                ▼
        ┌───────────────┐  否（无命中）   ┌──────────────┐
        │ 命中某节点?    ├───────────────►│ UNKNOWN 兜底  │
        └───────┬───────┘                 └──────┬───────┘
                │ 是                              │ 累计计数/解卡
                ▼                                 ▼
        handle_state / _dispatch_state 分支    等待下一帧
        按节点点击 last_pos(+offset)
```

## 五、如何扩展一个新节点

按项目约定（`.cursor/rules/project-structure.mdc`）：

1. `detector.py` 的 `GameState` 加枚举；
2. `templates_cn/` 或 `templates_global/` 放该画面的模板小图（文件名 = 模板名）；
3. `detector.py` 的 `checks` 列表加 `(模板名, GameState)`，**注意插入位置即优先级**；必要时在 `thresholds` 设专属阈值；
4. `core/gui/worker.py` 的 `AutomationWorker._dispatch_state`（及需要时 `main.py` 的 `handle_state`）加对应 `elif` 分支处理点击；
5. 坐标 / 偏移写进 `config.json` 的 `click_points` / `click_offsets`。
